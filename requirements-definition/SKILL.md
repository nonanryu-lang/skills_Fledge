# 要件定義書作成スキル

Webアプリケーション開発のヒアリングから要件定義書（Markdown）を作成する。

## ワークフロー判定

ユーザーの入力に応じて適切なフローを選択する：

- **対話形式ヒアリング** → 情報が少ない・ゼロから始める場合。ステップ1へ進む
- **一括入力からの整理** → メモ・議事録・要望リストなど既存情報がある場合。ステップ3へ進む

## ステップ1: 初期ヒアリング

まずプロジェクトの全体像を把握する。以下を確認する：

1. プロジェクトの背景と目的（何を解決したいか）
2. 対象ユーザー（誰が使うか）
3. 主要な機能のイメージ（何ができるべきか）

質問は1回につき2〜3問に抑え、ユーザーを圧倒しない。回答に応じて深掘りする。

**初回の質問例：**
```
このプロジェクトについて教えてください：
1. どのような課題を解決するためのアプリケーションですか？
2. 主に誰が使いますか？（社内ユーザー、顧客など）
3. 最も重要な機能を3つ挙げるとしたら何ですか？
```

## ステップ2: 詳細ヒアリング

初期ヒアリングの回答をもとに、不足情報を段階的に確認する。  
詳細なヒアリング項目は [references/hearing-guide.md](references/hearing-guide.md) を参照する。

**ヒアリングの進め方：**
- プロジェクト規模に応じてヒアリング項目を取捨選択する
- 小規模：概要・業務要件・機能要件を中心に確認
- 大規模：非機能要件・外部連携・運用要件も網羅する
- 回答が曖昧な場合は具体例を提示して確認する  
  例：「認証方式について、SSO連携が必要ですか？それともメール/パスワードで十分ですか？」
- 各カテゴリの確認後「他に気になる点はありますか？」と聞く
- 情報が十分に集まったらステップ3へ進む

## ステップ3: 要件定義書の作成

収集した情報を要件定義書にまとめる。テンプレートは [references/document-template.md](references/document-template.md) を参照する。

**作成方針：**
1. テンプレートの全セクションを使う必要はない。ヒアリング内容に応じて取捨選択する
2. 機能要件にはID（F-001等）を付与し、一意に識別できるようにする
3. 未確定事項は `[TBD]` と記載し、後続フェーズで決定する旨を明記する
4. 優先度（必須/重要/任意）を明示する
5. 出力ファイル名は `requirements-[プロジェクト名].md` とする

**一括入力の場合の追加手順：**
- 入力情報を読み解き、テンプレートの各セクションにマッピングする
- 情報が不足しているセクションを洗い出し、ユーザーに追加確認する
- 明記されていないが推測できる要件は記載した上で「◯◯という理解で合っていますか？」と確認する

## ステップ4: レビューと修正

要件定義書を出力した後：

1. 「内容を確認してください。修正・追加したい点はありますか？」とユーザーに確認する
2. フィードバックを反映して更新する
3. 最終版を出力する
